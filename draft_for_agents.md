# とにかく大事にしたい、全体の方針

## 勝手に、一気に、知らないうちに、評価できない、という実行は許されない

- 勝手に→コンテキストが十分でない、もしくはユーザーとの合意がない状態
- 一気に→必ずユーザーが評価できてかつ、最小で動く（テストできる）単一動作か、大きくても垂直スライスの最小単位を実装し、内容をユーザーに確認してもらい、合意をとるべき
  - もしこれが合意できれば、apply-patternしても良いという流れにしたい
- 知らないうちに→スコープ外、サイレントにすること
- 評価できない→ユーザーが何を確認したらいいか明確でない

## セッションフロー

- 明確な型はない方向性。
- ただし、エージェントができる限り能動的な提案やスキルトリガーが起きるようにしたい
- 議論を中心としたい。
- エージェントはクエスチョンツールや、mcp、websearchなども能動的に使って欲しい

## ハーネスなど

- 〇〇を読めとか、〇〇をやれという指示をagents.mdやプロンプトに書いてもエージェントは無視しがち
- なので、明確になるようにスキルやプラグインを使いたい
- 特にプラグインで強制的にコンテキスト注入や、コミット前にルールを注入してチェックさせるなどが重要。
  - プラグインは、必ず起きるとわかっているフロー上のタイミングを条件化できて、かつコントロールすることが何かあれば積極的に使いたい方針。

上記を満たすような、ハーネスやフローを構築したい。

## それぞれの項目についてのメモ

- discuss → setup ?
  - セッション冒頭に使うイメージ、できればエージェントがトリガー
  - エージェントは適宜ユーザーの意図を理解しながら、合意を取るためにクエスチョンツールや情報を集めて、ユーザーに質問、議論した上でゴールとゲートを確定
  - その他、開始前にgitなどを確認して、場合によってstashしたりコミットしたりしてクリーンにする工程あり
- agents.md
  - How a session worksどうのこうのみたいな案は良いと思う。その後もうちょっとブラッシュアップしたい
  - project-metaの項目をいくつか入れたい。headingはmetaとかかな。project-metaは削除方針。
  - agents.mdはメタ的な情報と、現状ある項目を構造的に整理して配置。
- implement-系
  - 統合したい。testは分離したほうがいいかな。それ以外はimplementとして1つのスキルにしapiとか各項目を羅列して、重複なども適宜整理する方針。
  - p-metaの文言を一部移植予定
  - この中に、implement前にまずユーザーに確認する、もしくは不足した情報がないか、懸念がないかについて確認し、適宜実行前にコンテキストを足す工程を入れる。これは上記の各項目において、色々違いがあるはずなので、各項目で例や定義を書く
  - ということは、事前のplanはいらない可能性あり？コンテキストが十分あるなと思ったら、このスキルをトリガーしたら自然に不測情報を再度聞く流れにすれば。しかも最小実行とアプライパターンを定義できれば。
    - もしくは、issueをどう扱うか、も合わせて議論がいるかも。
- issue
  - 簡単なものは特にissueを作る必要はないと思っているし、別に粒度もめちゃくちゃ大きいとかでなければ良い。1issue 1機能みたいな制限はなくて、意味があるまとまりならそれでいい
  - ただし、上記で書いたように、これをどう扱ってpjを管理するかがいまいちしっくりくるものがない。
  - なぜなら、以前はsemi-issue-drivenみたいなことやってたけど、issueがpj進行中に相当回数書き直す、削除する、新しく作るなどが起きすぎて、管理がだるすぎた。
